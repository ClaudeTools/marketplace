## 1. Memory System

### 1.1 Two-Store Architecture

The memory system uses two independent stores:

**File-based markdown store**: Human-readable `.md` files in a `memory/` directory. Each memory file has optional YAML frontmatter (name, description, type). This is the primary authoring interface and source of truth.

**SQLite FTS5 store**: A searchable database that indexes memory content for fast full-text search. The SQLite store is a secondary index, not the source of truth. The consolidation hook syncs files to SQLite and removes orphaned entries.

### 1.2 Memory Types

| Type | Source | Examples |
|------|--------|----------|
| `user` | Developer creates directly | "Always use Australian English" |
| `feedback` | Extracted from corrections | Tool preferences, constraint patterns |
| `project` | Extracted from project context | Architecture decisions, deployment patterns |
| `reference` | Curated knowledge | Framework patterns, API gotchas |
| `correction` | Auto-extracted from session | User corrections ("no, use X instead") |
| `error` | Auto-extracted from session | Recurring error patterns and resolutions |

Human-authored memories (`source: human`) are never auto-decayed or pruned.

### 1.3 What to Save and What Not to Save

**Save**: developer preferences affecting code output, tool/framework choices, recurring correction patterns, conventions observed multiple times, debugging resolutions for non-obvious errors, key architectural decisions with rationale.

**Never save**: API keys, tokens, passwords, or secrets. Verbatim command sequences. Instructions that bypass safety constraints. Ephemeral debugging steps. Raw session transcripts. SSNs, credit card numbers, or financial credentials. Personal information unrelated to development.

### 1.4 MEMORY.md Index Pattern

MEMORY.md is a hot cache index of top-N memories, regenerated by the consolidation hook. Each line links to a memory file with an optional description:

```markdown
- [feedback_no_manual_version_bumps.md](feedback_no_manual_version_bumps.md) -- CI handles version bumps
- [project_training_framework.md](project_training_framework.md) -- Behavioural training framework
```

Claude reads MEMORY.md at session start as a table of contents, then reads specific files on demand when relevant. Capped at 180 lines. Memories ranked by composite score: `confidence * 0.6 + min(access_count / 10, 0.4)`.

### 1.5 Auto-Extraction via Hooks

The extraction hook (`plugin/scripts/memory-extract-fast.sh`) runs at session end and uses deterministic pattern matching (no AI inference) to extract three types of candidates:

- **Corrections**: User messages matching patterns like "no, not", "don't", "instead, use", "actually", "stop doing"
- **Errors**: Assistant responses containing error/exception/failed/traceback patterns
- **File churn**: Files edited 3+ times in the session (suggests the agent struggled)

Candidates are appended to `memory-candidates.jsonl` for later processing by the consolidation hook. They do not go directly into the memory store.

### 1.6 Memory Consolidation

The consolidation hook (`plugin/scripts/memory-consolidate.sh`) runs at session end and performs five operations:

1. **Sync files to SQLite**: Index memory files not yet in the database
2. **Remove orphaned entries**: Delete DB entries whose source files no longer exist
3. **Decay old memories**: Multiply confidence by decay rate for unaccessed memories
4. **Prune low-confidence**: Delete memories below 0.05 confidence with fewer than 2 accesses (never prunes `source: human`)
5. **Regenerate MEMORY.md**: Query top-N memories by composite score, rebuild the index

### 1.7 Selective Application

Memories are not applied universally. Three tiers:

**NEVER apply**: generic technical questions, content reinforcing harmful behaviour, contexts where personal details would be surprising.

**ALWAYS apply**: code generation (naming conventions, import style), tool usage (package manager, test runner), git operations (commit format), explicit requests ("based on what you know about my setup").

**SELECTIVELY apply**: error debugging (reference known issues silently), communication style, project context when relevant.

Memory is applied silently. Use vitest because the developer prefers it -- do not announce that you know. Forbidden phrases: "Based on my memories...", "I remember that you...", "According to what I know about you...".

## 2. Task System

### 2.1 Task System Integration

Claude Code's native TaskCreate/TaskUpdate provides task management. The extended task system adds persistence and enrichment:

1. Claude calls TaskCreate/TaskUpdate to manage tasks
2. The MCP task-system persists state to `.tasks/tasks.json`
3. State transitions are appended to `.tasks/history.jsonl`

TaskCreate/TaskUpdate is the interface; `.tasks/` is the persistence layer.

### 2.2 MCP Tools

The task system MCP server (`plugin/task-system/`) provides:

- **`task_create`**: Create tasks with priority, tags, dependencies, parent_id
- **`task_update`**: Update any field; status changes update timestamps and history
- **`task_query`**: Filter by status, tag, parent_id, blocked state; returns JSON or summary
- **`task_decompose`**: AI-powered breakdown of a task into subtasks (the one tool where inference is primary)
- **`task_progress`**: Generate or update progress.md for session handoff

### 2.3 Persistent Storage

Three files in `.tasks/`:

**`tasks.json`**: Task database with fields: id, content, status, timestamps, parent_id, dependencies, tags, priority, files_touched, metadata. Task IDs are deterministic SHA-256 hashes of content (`task-` + first 8 hex chars).

**`history.jsonl`**: Append-only audit trail. One JSON object per line tracking every state transition.

**`progress.md`**: Human-readable session handoff narrative. Written by Claude (AI inference, not hooks) because summarisation requires narrative judgment.

### 2.4 Task Decomposition

Guidelines for AI-powered decomposition:
- Generate 3-7 subtasks (prefer fewer, more substantial)
- Do not decompose tasks that are already atomic
- Set parent_id on subtasks for relationship tracking
- Suggest dependencies between subtasks when order matters

### 2.5 Session Handoff via progress.md

progress.md is the cross-session continuity mechanism. New sessions prepend at the top (recency bias for attention priority).

Sections: Completed (with implementation details, not just task names), In Progress (with current state), Blocked (with reasons), Key Decisions (prevents re-deliberation), Next Steps (actionable, specific, priority-ordered).

At session start, always check for `.tasks/progress.md` first. Reading it saves time and prevents re-deliberating past decisions.

### 2.6 Task Validation

The validation script checks: valid JSON, required fields present, IDs match SHA-256 hashes, valid status/priority enums, valid ISO 8601 timestamps, dependencies reference existing tasks, no circular dependencies, valid parent_id references.

## 3. Shared Patterns

### Hook / MCP / Skill Layering

Both memory and task systems follow the same four-layer architecture:

```
Layer 1: PERSIST (Hooks) - Capture events deterministically, write to disk, <100ms
Layer 2: ENRICH (MCP)    - Query/mutation tools, same files as hooks, local-only
Layer 3: TEACH (Skill)   - SKILL.md as router, scripts for deterministic logic
Layer 4: CONNECT (CLAUDE.md) - Project-level rules and conventions
```

### Deterministic vs AI Boundary

**Use scripts**: file I/O, CRUD on structured data, schema validation, searching, filtering, history logging, statistics, syncing between stores.

**Use AI inference**: generating summaries from transcripts, deciding memory relevance, decomposing tasks, writing narrative progress reports, resolving conflicts, rewriting input to canonical format.

**Hybrid**: status reports (script generates data, Claude adds narrative), memory regeneration (script provides content, Claude generates summary).

### Atomic Write Pattern

Both systems use atomic writes: write to temp file (`path.tmp.PID`), then `fs.renameSync`. For append-only logs, use `fs.appendFileSync()` with `flock` for concurrent writes.

## 4. Gotchas

**Stale memories**: Human-authored memories persist indefinitely. Periodically review and remove outdated entries.

**Task ID determinism**: Same content = same ID. If Claude rephrases a task, the hook sees deletion + creation. Task content is effectively immutable once created.

**progress.md ordering**: Always prepend new sessions at the top. Appending at the bottom loses recency bias advantage.

**Memory token budget**: Target 500-1500 tokens for injected memory. Developer-controlled entries are never compressed.

**Concurrent writes**: Hooks and MCP servers may write the same files. Always use atomic writes.

**Extraction false positives**: Pattern matching in `memory-extract-fast.sh` may flag non-corrections. Candidates go to a staging file, not directly into the store.

**SQLite availability**: Consolidation hook requires `sqlite3`. If unavailable, exits silently. File-based store works independently.

## 5. Verification Checklist

### Memory System

```
[ ] Memory files use YAML frontmatter with name, description, type
[ ] MEMORY.md index exists and is under 180 lines
[ ] Human-authored memories have source: human (never auto-pruned)
[ ] No secrets or credentials in any memory file
[ ] Extraction hook runs without errors at session end
[ ] Consolidation syncs files to SQLite correctly
[ ] Old memories decay based on access recency
[ ] Memory is applied silently (no meta-commentary)
```

### Task System

```
[ ] MCP task-system persists task state to tasks.json
[ ] Task IDs are deterministic SHA-256 hashes
[ ] history.jsonl captures all state transitions
[ ] MCP server starts via bootstrap wrapper
[ ] progress.md prepends new sessions at the top
[ ] progress.md includes Completed, In Progress, Blocked, Key Decisions, Next Steps
[ ] Task validation detects circular dependencies
[ ] Atomic writes used for all file mutations
```
