**Difficulty: Beginner**

:::note[Prerequisites]
- [claudetools installed](../getting-started/installation.md) — plugin active in Claude Code
- A git repository — run `git init` in your project directory if you don't have one yet
:::

Get claudetools working in a new or existing project — from install through first session, with indexing, session dashboard, and a CLAUDE.md that captures project conventions.


## What you need
- claudetools installed globally via the marketplace
- A git repository (existing project or `git init` a new one)

## Steps

### 1. Verify installation

Open a Claude Code session in your project directory and confirm the plugin is active:

```
/session-dashboard
```

The session dashboard should appear showing plugin status, session mode, and active tools. If it does not appear, install or reinstall claudetools from the marketplace.

### 2. Wait for automatic indexing

On every session start, claudetools runs `session-index.sh` in the background to build the codebase-pilot index. For a large project this takes 10–30 seconds. You can check when it finishes:

```bash
node ${CLAUDE_PLUGIN_ROOT}/codebase-pilot/dist/cli.js map
```

If the map returns results, indexing is complete. If it returns nothing, wait a moment and try again.

### 3. Explore the project structure

Run a map to understand what you are working with:

```
explore this codebase
```

Claude uses codebase-pilot to generate an overview of languages, directory structure, entry points, and key exports. Read the output — it becomes the foundation for all subsequent work in this project.

### 4. Check the session dashboard

```
/session-dashboard
```

The dashboard shows:
- **Session mode** — whether quiet mode is active
- **Active tasks** — any tasks in progress from a previous session
- **Plugin health** — which hooks and skills are loaded
- **Recent activity** — the last few actions taken

### 5. Configure quiet mode if needed

If Claude is being too verbose in this project (explaining every step, summarising after each action), enable quiet mode:

```
enable quiet mode
```

Quiet mode suppresses narration and summaries, leaving only results. Disable it with:

```
disable quiet mode
```

### 6. Add a CLAUDE.md file

A `CLAUDE.md` at the project root tells Claude how to behave in this specific project. Create one with the conventions you want enforced:

```markdown
# my-project

## Stack
- Node.js 20, TypeScript 5, Prisma ORM
- PostgreSQL (local dev via Docker)
- Next.js App Router

## Conventions
- Use `pnpm` not npm
- All DB migrations via `pnpm prisma migrate dev`
- Run tests with `pnpm test`
- TypeScript strict mode is required — no `any`

## Rules
- Never edit files in `generated/` — they are auto-generated by Prisma
- Always run `pnpm typecheck` before committing
```

Commit `CLAUDE.md` to the repository so every team member's Claude session picks up the same conventions.

### 7. Create your first task

With the project set up, create a task to track your first piece of work:

```
/managing-tasks new [description of your first task]
```

## What happens behind the scenes

- **session-index.sh** runs as a `PostSessionStart` hook — it indexes the project automatically without you asking
- **session-dashboard** reads plugin state from the hook registry and task system — it is a read-only view
- **CLAUDE.md** is loaded at session start and takes precedence over Claude's defaults — it is the highest-priority instruction source for project-specific rules
- The codebase-pilot index is stored locally (not committed) and rebuilt when the session starts

## Tips

- For a monorepo, create a `CLAUDE.md` at the root and additional ones in each package with package-specific rules
- If the index seems stale after adding many new files, ask Claude to reindex: "reindex the codebase"
- Quiet mode is per-session, not per-project — if you always want it on for this project, add `default: quiet mode on` to your `CLAUDE.md`
- The session dashboard is useful at the start of any session to see what tasks are pending from last time
